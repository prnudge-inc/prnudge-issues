# Contributing Guidelines

Thanks for taking the time to help improve PRNudge.

## How to contribute

### 1) Bug reports
Use the **Bug Report** template and include:
- Installation ID
- Repo full name
- Steps to reproduce (as minimal as possible)
- Expected vs actual behavior
- Time window and timezone
- Any relevant logs (redacted)

If we cannot reproduce the issue, we may ask for:
- A minimal reproduction repo (public)
- Additional logs (redacted)
- Confirmation of app permissions and installation scope

### 2) Feature requests
Use the **Feature Request** template and include:
- The problem youâ€™re trying to solve
- Proposed solution
- Alternatives considered
- Any constraints (org policies, compliance, repo size)

### 3) Questions / Support requests
If Discussions are enabled, prefer using Discussions for questions.
Otherwise, open an issue using the **Support Request** template.

## Labels & triage

We typically use:
- `triage` for new issues
- `bug` for confirmed defects
- `enhancement` for feature requests
- `needs-info` when more details are required
- `blocked` when waiting on external dependency

## Code of Conduct

Be respectful and constructive.
Harassment, abuse, or doxxing will not be tolerated.

## Pull requests

This repo is primarily for support. If we accept PRs:
- Keep changes small and focused
- Link to an issue
- Avoid introducing new dependencies unless justified
